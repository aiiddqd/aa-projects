# RFC: Визуальный редизайн MVP-лендинга Ads Art Group

- ID: RFC-260628-VISUAL
- Дата: 2026-06-28
- Статус: Draft
- Область: Визуал, контент-подача, UI-система

## 1. Контекст и проблема

Текущий одностраничник функционально собран, но визуально воспринимается как типовой корпоративный шаблон: безопасная сетка, знакомая daisyUI-подача, слабая уникальность бренда в первом экране.

Цель RFC: предложить визуальную систему, которую нельзя спутать с другим B2B-лендингом в индустриальной нише, без потери читаемости и конверсии.

## 2. Предмет, аудитория и единичная задача страницы

- Предмет: художественное промышленное брендирование (контейнеры, резервуары, фасады, спецтехника).
- Аудитория: маркетинг и операционные руководители промышленных компаний, девелоперы индустриальных площадок, собственники логистических парков.
- Единичная задача страницы: довести посетителя до прямого контакта (Telegram, email, телефон) через доверие к масштабу и управляемости процесса.

## 3. Двухпроходный дизайн-план

### Проход 1: токены, структура, сигнатура

#### 3.1 Color tokens (4-6 именованных цветов)

- Galvanized Light: #D8E1E8
- Zinc Plate: #A7B5C2
- Asphalt Ink: #1F2A33
- Safety Orange: #F05A28
- Cobalt Mark: #1251A3
- Survey Lime: #B7D85B

Назначение:
- База страницы не белая, а "оцинкованная" светло-холодная.
- Контраст и акценты строятся на паре Safety Orange + Cobalt Mark.
- Survey Lime используется дозированно для чисел и технических ярлыков.

#### 3.2 Type system (роли шрифтов)

- Display: Bebas Neue (только для H1/H2 и крупных чисел).
- Body: IBM Plex Sans (основной текст, интерфейсные подписи).
- Utility/Data: IBM Plex Mono (бейджи, шаги процесса, метрики, микро-лейблы).

Принцип: выразительный display только в ключевых точках, остальная типографика максимально инженерная и точная.

#### 3.3 Layout concept

Концепт: "Паспорт объекта".

Страница воспринимается как инженерный досье-лист: крупные поля, штампы, технические подзаголовки и визуальная траектория нанесения краски, проходящая через секции.

ASCII-варианты:

Вариант A (выбран):

```
+-----------------------------------------------------------+
| LOGO                NAV                             CTA   |
+-----------------------------------------------------------+
| HERO LEFT: THESIS                 HERO RIGHT: OBJECT CARD |
| Huge display line                 Metrics + status tags   |
| Supporting copy + actions                                  |
+------------------------/----------------------------------+
| SERVICES (card grid with technical labels)                |
+----------------------/------------------------------------+
| PROJECTS (task -> solution -> result columns)             |
+--------------------/--------------------------------------+
| PROCESS (linear production rail, not decorative timeline) |
+------------------/----------------------------------------+
| TEAM + CONTACT STRIP                                       |
+-----------------------------------------------------------+
```

Вариант B (отклонен): плотная газетная сетка с тонкими линиями.
Причина отказа: выглядит как общий тренд, а не как мир промышленных объектов бренда.

#### 3.4 Signature element

"Линия прохода": диагональная полоса, напоминающая траекторию промышленного нанесения, которая проходит через разделы и связывает их в единый маршрут проекта.

Это единственный смелый прием на странице. Остальные элементы держатся в строгой инженерной логике.

### Проход 2: ревизия на не-типовость

Проверка против шаблонных решений:

- Убрали "чисто белый SaaS-фон" -> заменили на материал "оцинкованный металл".
- Не ушли в типичный черный neobrutal -> выбрана дневная индустриальная палитра.
- Отказались от декоративной нумерации "01/02/03" там, где порядок не критичен.
- Зафиксировали один риск-элемент (диагональная линия прохода), чтобы не перегрузить дизайн.

## 4. Предлагаемые изменения по секциям

### 4.1 Hero как тезис

- Новый H1 с более характерной формулировкой:
  "Брендинг промышленных поверхностей, который видно до въезда на объект".
- Справа не абстрактные stats, а "карта объекта": тип объекта, площадь, срок, режим работ.
- Подзаголовок описывает управляемость: бриф -> превиз -> нанесение -> сдача.

### 4.2 Services

- Карточки услуг становятся "операционными модулями":
  - Что делаем
  - На каких поверхностях
  - Ограничения/условия
- Каждый модуль получает короткий технический ярлык в моноширинном стиле.

### 4.3 Projects

- Сценарий каждой карточки в одном формате: Контекст -> Решение -> Результат.
- Результат всегда количественный (м2, сроки, эффект на операцию/видимость).

### 4.4 Process

- Вместо декоративного таймлайна: производственная "рельса" из 5 стадий.
- Каждая стадия отвечает на вопрос заказчика: что я получаю на этом шаге.

### 4.5 Contacts

- Контактный блок делает один выбор простым: "Выберите удобный канал, мы ответим в течение N времени".
- Кнопки визуально дифференцируются по приоритету, но без визуального шума.

## 5. Motion и доступность

- Motion-фокус только один: плавное появление секций по мере скролла + движение "линии прохода".
- Поддержка prefers-reduced-motion: отключение всех несущественных анимаций.
- Видимый keyboard focus для всех интерактивных элементов.
- Мобильный приоритет: упрощение композиции, сохранение сигнатуры в легкой версии.

## 5.1 Промежуточное решение: emoji вместо иконок/иллюстраций

Пока нет собственной системы иконок и иллюстраций, допускается управляемое использование emoji как временного визуального слоя для повышения считываемости блоков.

Принципы применения:

- Emoji используются как маркеры смысла, а не как декор.
- Один смысловой блок = один emoji; не использовать цепочки из 2-3 символов подряд.
- Не использовать emoji в основном H1 и в ключевых CTA-кнопках, чтобы не удешевлять подачу.
- Для одного типа сущности использовать один и тот же emoji по всей странице для консистентности.
- Ограничение насыщенности: не более 1 emoji на 120-180 символов текста в контентных абзацах.

Рекомендуемые зоны:

- Заголовки карточек услуг и кейсов.
- Микро-лейблы в процессных шагах.
- Подписи к метрикам и статусам в hero-правой панели.

Нежелательные зоны:

- Навигационное меню.
- Юридические/контактные строки в футере.
- Служебные сообщения об ошибках и валидации.

Семантический набор для MVP:

- 🏗️ инфраструктура и объект
- 🎨 художественная часть и графика
- 🚚 спецтехника и транспорт
- 🧪 технология, материалы, тесты
- 📍 навигация и маркировка
- ⏱️ сроки и этапы
- 📞 контакт и быстрый выход на связь

Техническая реализация:

- Emoji добавляются в текстовые узлы и могут получать отдельный span-класс для выравнивания по базовой линии.
- Для стабильного рендера на платформах использовать системные emoji-шрифты без внешних ассетов.
- После появления собственной икон-системы emoji удаляются поэтапно в рамках отдельной задачи UI-hardening.

## 6. План внедрения (Astro)

### Фаза 1: токены и тема

- Ввести кастомную тему вместо стандартной corporate.
- Задать CSS custom properties для color/type/spacing.
- Подключить шрифты через Layout и ограничить display-шрифт только заголовками.

Затрагиваемые файлы:
- astro/src/styles/global.css
- astro/src/layouts/Layout.astro

### Фаза 2: композиция первого экрана

- Пересобрать hero в двухпанельный "тезис + карта объекта".
- Добавить сигнатурную "линию прохода" (CSS + псевдоэлемент).

Затрагиваемые файлы:
- astro/src/pages/index.astro
- astro/src/styles/global.css

### Фаза 3: карточки и процесс

- Стандартизировать карточки услуг/кейсов по единому контентному шаблону.
- Заменить timeline на производственную рельсу.

Затрагиваемые файлы:
- astro/src/pages/index.astro
- astro/src/styles/global.css

### Фаза 4: финальная шлифовка

- Проверка контрастности и фокус-состояний.
- Тест на 360px/768px/1280px.
- Уменьшение декоративных элементов на 15-20% перед релизом (правило "убрать один аксессуар").

## 7. Критерии приемки

- Первый экран узнается как страница индустриального арт-брендинга без чтения всего текста.
- Визуал не похож на стандартный daisyUI corporate-шаблон.
- Остается высокая читаемость текста на мобильных устройствах.
- Все CTA визуально заметны и доступны с клавиатуры.
- Есть стабильная сигнатура (линия прохода), но без перегруза эффектами.

## 8. Риски и смягчение

- Риск: сигнатурная диагональ отвлечет от CTA.
  - Смягчение: ограничить насыщенность и движение, оставить ее на заднем слое.
- Риск: display-шрифт снизит читаемость на малых экранах.
  - Смягчение: clamp-скейл и строгие лимиты на длину заголовков.
- Риск: визуальная новизна ухудшит скорость.
  - Смягчение: без тяжелого видео в первом релизе, только CSS-эффекты и оптимизированные изображения.

## 9. Решение для старта

Рекомендуется реализовать Фазы 1-2 в ближайшем спринте как минимальный визуальный рывок, затем оценить CTR главного CTA и глубину скролла перед переходом к Фазам 3-4.
